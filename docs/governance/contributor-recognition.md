# Contributor recognition policy

## Principle

Useful feedback is project work. ChatCommons treats people who identify product,
design, accessibility, translation, security, testing, or documentation issues
as contributors when they want that recognition.

## Feedback close-out

For every actionable annotation:

1. acknowledge the feedback;
2. describe what changed or why it will not be implemented;
3. move implemented work to `client_review`, not directly to `completed`;
4. wait for acceptance or an explicit closure rule;
5. ask through the review toolbar whether the reviewer wants public credit and
   which name or link to use. The submission enters the existing owner inbox as
   a normal, withdrawable feedback record; it does not publish automatically.

The default bilingual acknowledgement is intentionally editable. A specific
reply is preferred when it can name the change that was made.

## Privacy

- Public credit is opt-in.
- Never infer consent from a screenshot, private message, or review token.
- Store only the preferred public name, contribution area, and optional link.
- Allow a contributor to change or remove their credit later.
- Do not expose reviewer credentials, private feedback screenshots, IP
  addresses, or internal audit records in the contributors list.

## Scope

`CONTRIBUTORS.md` is the project-wide human-readable list. Git history remains
the authoritative record for commits, while the contributors list recognizes
valuable work that does not produce a commit.
